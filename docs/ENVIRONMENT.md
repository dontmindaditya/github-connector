# Environment Variables

Reference for every variable the connector reads. All are validated at boot by `src/lib/env.ts` — a missing or malformed value crashes the process with a readable error before any request is served.

## Quick reference

| Variable | Required | What it is |
|---|---|---|
| `DATABASE_URL` | yes | PostgreSQL connection string |
| `APP_URL` | yes | Public URL of your deployment (no trailing slash) |
| `NODE_ENV` | no | `development` / `test` / `production` (defaults to `development`) |
| `GITHUB_APP_ID` | yes | Numeric App ID |
| `GITHUB_APP_SLUG` | yes | URL slug of your App |
| `GITHUB_CLIENT_ID` | yes | OAuth Client ID |
| `GITHUB_CLIENT_SECRET` | yes | OAuth Client Secret |
| `GITHUB_PRIVATE_KEY` | yes | PEM-formatted RSA private key |
| `GITHUB_WEBHOOK_SECRET` | yes | HMAC secret for webhook verification (min 16 chars) |
| `ENCRYPTION_KEY` | yes | base64-encoded 32-byte AES-256 key |

## Database

### `DATABASE_URL`

Standard PostgreSQL URL. Works with any provider.

```
DATABASE_URL="postgresql://USER:PASS@HOST:PORT/DBNAME"
```

**Provider examples:**

```bash
# Local Docker
DATABASE_URL="postgresql://postgres:postgres@localhost:5432/github_connector"

# Neon (serverless Postgres)
DATABASE_URL="postgresql://user:pass@ep-cool-name.us-east-1.aws.neon.tech/dbname?sslmode=require"

# Supabase
DATABASE_URL="postgresql://postgres:pass@db.xyz.supabase.co:5432/postgres"

# Vercel Postgres — set automatically when you add the integration
```

**Serverless tip.** On Vercel each function invocation can open a new connection. If your provider supports it, add `?pgbouncer=true&connection_limit=1` to the URL to use connection pooling. Neon and Supabase both expose pooler endpoints; use those for `DATABASE_URL` in production and the direct endpoint only for migrations (`DIRECT_URL` in `schema.prisma` if you split them).

## App

### `APP_URL`

The public URL of your deployment. Used to build redirect targets after the GitHub install flow.

```
APP_URL="http://localhost:3000"            # dev
APP_URL="https://your-app.vercel.app"      # Vercel
APP_URL="https://github-connector.example" # custom domain
```

**No trailing slash.** The code concatenates paths directly.

### `NODE_ENV`

Optional. Defaults to `development`. The build sets this to `production` for you on most hosts. Affects:

- Prisma query logging verbosity (more in dev)
- Cookie `Secure` flag (only set in production)

## GitHub App

See [GITHUB_APP_SETUP.md](./GITHUB_APP_SETUP.md) for where each value comes from.

### `GITHUB_APP_ID`

The numeric App ID shown at the top of your App's settings page on github.com. NOT the Client ID.

```
GITHUB_APP_ID="123456"
```

Used as the `iss` (issuer) claim when minting App JWTs.

### `GITHUB_APP_SLUG`

The slug portion of your App's public URL.

```
# Public URL: https://github.com/apps/my-connector
GITHUB_APP_SLUG="my-connector"
```

Used to build the install redirect: `https://github.com/apps/{slug}/installations/new?state=...`.

### `GITHUB_CLIENT_ID`

OAuth Client ID from the App settings page. Format: `Iv1.xxxxxxxxxxxxxxxx`.

```
GITHUB_CLIENT_ID="Iv1.abc123def456"
```

The connector doesn't currently run the user-OAuth flow (we use the App JWT instead), but having it set keeps the door open for adding "Sign in with GitHub" later.

### `GITHUB_CLIENT_SECRET`

Generated by clicking "Generate a new client secret" on the App settings page. Shown ONCE — copy immediately.

```
GITHUB_CLIENT_SECRET="<40-char-hex-string>"
```

Treat as a secret. If you lose it, generate a new one and update env everywhere.

### `GITHUB_PRIVATE_KEY`

The App's RSA private key in PEM format. Used to sign the JWT we present to GitHub.

**Local dev:**

```
GITHUB_PRIVATE_KEY="-----BEGIN RSA PRIVATE KEY-----
MIIEpAIBAAKCAQEA...
...many lines...
-----END RSA PRIVATE KEY-----"
```

**Vercel and any host that rejects multi-line env values** — convert newlines to literal `\n`:

```bash
awk 'NF {sub(/\r/, ""); printf "%s\\n",$0;}' your-app.private-key.pem
```

Paste the single-line output as the env value. `src/lib/env.ts` runs `.replace(/\\n/g, "\n")` at boot to restore real newlines.

**Verifying it loaded correctly** — the env loader will throw if the key can't be parsed by Node's crypto module. If you see "PEM_read_bio_PrivateKey failed" or similar at boot, the key string is mangled.

### `GITHUB_WEBHOOK_SECRET`

Shared secret for HMAC-SHA256 webhook signature verification. Must match exactly what you set on the App's webhook configuration page.

Generate:

```bash
openssl rand -hex 32
```

```
GITHUB_WEBHOOK_SECRET="64-hex-chars-from-openssl"
```

Minimum 16 characters (enforced by the env loader). If the secret here doesn't match the App's webhook config, every incoming webhook returns 401 and GitHub stops retrying after 8 attempts.

## Encryption

### `ENCRYPTION_KEY`

Master key for AES-256-GCM. Used to encrypt cached installation tokens (and any other secret you add later via `encrypt()`).

**Must decode to exactly 32 bytes.** The env loader validates this on boot.

Generate:

```bash
openssl rand -base64 32
```

```
ENCRYPTION_KEY="44-char-base64-string-from-openssl"
```

### Rotation

When you rotate `ENCRYPTION_KEY`:

- All cached `EncryptedToken` rows become undecryptable. That's harmless — they're cache; the next API call mints a fresh installation token and writes a new encrypted row.
- Any other long-lived encrypted data (if you add some) becomes permanently lost. Plan rotations with a key-versioning scheme (`encryption_keys` table with `key_id` columns on encrypted rows) if you store anything you can't re-derive.

For routine rotation, the safe procedure is:

```bash
# 1. Generate the new key
openssl rand -base64 32

# 2. Set ENCRYPTION_KEY to the new value in your secret manager
# 3. Deploy
# 4. Wipe the cache table — saves the auto-refresh dance on first hit
psql "$DATABASE_URL" -c "DELETE FROM encrypted_tokens;"
```

## Generating all secrets in one shot

```bash
cat <<'EOF'

# Paste into .env.local:
ENCRYPTION_KEY="$(openssl rand -base64 32)"
GITHUB_WEBHOOK_SECRET="$(openssl rand -hex 32)"
EOF
```

## How validation works

The loader (`src/lib/env.ts`) parses `process.env` with Zod the first time anything imports it. Failure raises a single error listing every problem:

```
Error: Invalid environment variables:
  - GITHUB_APP_ID: Required
  - ENCRYPTION_KEY: ENCRYPTION_KEY must decode to exactly 32 bytes (got 16)
```

This is intentional — refusing to start beats crashing on the first request with a cryptic stack trace.

Two extra checks run after Zod:

1. `ENCRYPTION_KEY` is base64-decoded and asserted to be exactly 32 bytes.
2. `GITHUB_PRIVATE_KEY` has `\n` literals converted back to real newlines.

After that, every other module imports `{ env }` from this file instead of touching `process.env` directly. That way TypeScript knows the values are present and non-nullable.

## Variables you DON'T need

- `GITHUB_TOKEN` — we never use a static GitHub token. Authentication is per-installation, dynamic.
- `JWT_SECRET` — we sign App JWTs with `GITHUB_PRIVATE_KEY`, not a separate secret.
- `SESSION_SECRET` — depends on your auth provider; not used by the connector itself.
